Classes

SessionM

Session M service interface. This is main class in Session M iOS SDK. It defines methods for all main operations such logging an action and displaying an achievement or a user portal, etc.

Basic API usage

Following is the simplest way of integrating SessionM service within your application code. It assumes that you have registered your application in Session M developer portal and configured achievements.

  1. Start the service by calling SMStart macro.

  2. Log action in appropriate places in your code log user action by calling SMAction. When new achievement is earned it will be displayed automatically using the style chosen in the developer portal.

When minimal API described before is not sufficient application can obtain SessionM instance by calling sharedInstance and, optionally, implementing and registering SessionMDelegate instance.

For example, to override automatic achievement display do the following:

  1. Implement SessionMDelegate and, specifically, sessionM:shouldAutopresentAchievement: method to get notified that new achievement is earned and is ready for display. Return NO to suppress achievement display.

  2. At appropriate place in your application call presentActivity: to display the achievement UI.

UI Activity

Application user engages with SessionM service via UI activity or flow. UI activity starts with an initial view which can be partial- (e.g. achievement alert) or full-screen (e.g. user portal). A UI activity can be comprised on multiple views. Partial-screen view can transition to full-screen view.

Application is notified by SessionM UI display by implementing the following methods of SessionMDelegate: sessionM:willPresentActivity:, sessionM:didPresentActivity:, sessionM:willDismissActivity: and sessionM:didDismissActivity:.

To present the UI SessionM adds its UIView and/or UIViewControllers to application's view and view controller hierarchy. By default, the following algorithm is to find appropriate superview or presenting controller:

  1. Traverse UIViewController chain starting with root view controller and, if set, find top-most UIViewController object.

  2. If view controller cannot be determined in step 1, iterate through UIWindow subviews finding the top-most visible UIView. If not found, use key UIWindow object.

When SessionM presents a UIView it adds it to the UIView found in steps above. When SessionM presents a UIViewController it uses the controller found in step 1 or, if not available, adds its view to the view found in step 2.

Application can instruct SessionM to use specific view and controller objects by implementing the following SessionMDelegate methods: sessionM:viewForActivity: and sessionM:viewControllerForActivity:.

Special consideration is taken when presenting SessionM UI activity when keyboard is displayed - the activity display is delayed until keyboard is dismissed. This applies to both when activity is auto-presented as a result of action logging or when application is explicitly presenting it by invoking presentActivity: method.

Achievement UI Customization

Application can customize achievement presentation to suit its style and UI flow by implementing the following steps:

  1. Configure achievements as "custom" in the development portal

  2. Obtain an achievement data object by calling unclaimedAchievement property and, optionally, implementing sessionM:didUpdateUnclaimedAchievement: delegate method to get notified about available achievements.

  3. Extend SMAchievementActivity class or use its instance as a callback object in conjunction with SMAchievementData object to present the achievement UI and notify SessionM about relevant lifecycle events.

For more information about using custom achievement API see Development Guide.

Session state

SessionM service, once started, is operational regardless of whether network connection is available or not. In online (connected) state the activity UI is available for display. In offline (disconnected) state actions logged through the service are captured but UI is not available. Upon re-connection the actions are synchronized with SessionM service. Application can be notified by SessionM state transitions via sessionM:didTransitionToState: delegate callbacks. SessionM service is stopped when application is backgrounded and re-started upon foregrounding.

Threading

Almost all SessionM API methods are asynchronous. Implement and register SessionMDelegate to get notified about the outcome of the method calls and other SessionM service events. Delegate calls are made on the main thread.

Location capture

To capture the location information you will need to start CCLocationManager and register for its notifications as described in Apple Location Awareness Programming Guide. As part of location notification processing update SessionM locationCoordinate property.

Logging

To diagnose problems and ensure proper SDK integration SessionM service implements basic logging which can be controlled using logLevel property. For basic output capturing all major SDK events use SMLogLevelInfo level. For more detailed, low level output use SMLogLevelDebug level. It is highly recommended to capture and provide debug output when reporting a problem Session M technical support.

SessionMState

Session state enum. 

typedef enum SessionMState {  
    SessionMStateStopped,  
    SessionMStateStartedOnline,  
    SessionMStateStartedOffline 
} SessionMState;
Fields
SessionMStateStopped

SessionM service is stopped.

SessionMStateStartedOnline

SessionM service is started and is working in online state, i.e. connected to the network.

SessionMStateStartedOffline

SessionM is started and is working in offline state, i.e. without network connection. Unlike online state, some operations, e.g. activity display may not be available in this state. SessionM tries to reconnect automatically and will transition to SessionMStateStartedOnline state once it has successfully been able to do so.
